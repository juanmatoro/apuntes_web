1Clases tipadas

Una clase es una plantilla para crear objetos con la misma forma. TypeScript te ayuda a definir desde el inicio qué datos y qué tipos tiene cada instancia.

  • Propiedad: dato que guarda estado (por ejemplo, nombre).
  • Método: acción de la clase (por ejemplo, getPrecio()).
  • Constructor: función que se ejecuta al hacer new.
class Producto {
  constructor(
    public sku: string,
    public nombre: string,
    private precio: number
  ) {}

  getPrecio(): number {
    return this.precio;
  }
}

Lectura rápida del ejemplo

  • public sku y public nombre se pueden leer desde fuera.
  • private precio solo puede usarse dentro de la clase.
  • La clase expone el precio de forma controlada con getPrecio().

Relación con el proyecto de clase

En demo.-js/13-constructor.js y demo.-js/16-ES6-herencia.js se ve que la versión con class es la forma moderna de expresar el patrón constructor.

class User {
  constructor(
    public name: string,
    public age: number = 22,
    public role: "admin" | "user" = "user"
  ) {}
}

2Modificadores de acceso: cuándo usar cada uno

Los modificadores en TypeScript definen la frontera de tu API de clase. La regla general es: exponer poco, encapsular lo inestable y dejar claro qué puede cambiar.

public (API externa)

Úsalo para propiedades o métodos que sí quieres que consuman otros módulos.

class UserService {
  public version = "1.0.0";

  public healthcheck(): boolean {
    return true;
  }
}

private (encapsulación interna)

Úsalo cuando una propiedad no debe tocarse desde fuera ni desde subclases.

class PasswordValidator {
  private minLen = 8;

  public validate(pass: string): boolean {
    return pass.length >= this.minLen;
  }
}

const v = new PasswordValidator();
// v.minLen; // Error TS: propiedad privada

protected (extensión por herencia)

Úsalo cuando el miembro debe ser invisible para consumidores, pero reutilizable por subclases.

class RepoBase {
  protected log(msg: string): void {
    console.log("[repo]", msg);
  }
}

class UserRepo extends RepoBase {
  save(): void {
    this.log("guardando usuario"); // OK
  }
}

// new RepoBase().log("x"); // Error TS

readonly (inmutabilidad de referencia)

Úsalo para datos que se fijan al crear la instancia y no deben reasignarse.

class Order {
  constructor(
    public readonly id: string,
    public total: number
  ) {}
}

const o = new Order("A-1", 100);
// o.id = "B-2"; // Error TS
o.total = 120; // OK

Criterio de diseño rápido

  • Empieza con private y abre a protected/public solo si hay un caso real.
  • Usa readonly para IDs, fechas de creación y configuración estable.
  • Si algo es public, trátalo como contrato: cambiarlo puede romper consumidores.

Importante: private (TS) vs #private (JS)

  • private de TS es validación de compilador.
  • #campo es privacidad real en tiempo de ejecución (como en demo.-js/14-class.js).
class Caja {
  #saldo = 0; // privacidad real de JS
  private etiqueta = "A"; // chequeado por TS

  ingresar(n: number): void {
    this.#saldo += n;
  }
}

Miembros estáticos y bloque static {}

Basado en el ejemplo del proyecto: usa estáticos para estado compartido por clase (contadores, factorías, configuración).

class User {
  static usersNumber = 0;
  static {
    console.log("Clase User cargada");
  }

  constructor(public name: string) {
    User.usersNumber++;
  }
}

3Herencia e interfaces

La herencia permite reutilizar una clase base. Las interfaces definen un contrato (qué debe poder hacer una clase), sin imponer implementación.

  • Usa extends cuando compartes comportamiento real entre clases.
  • Usa implements cuando solo quieres garantizar una forma común.
interface Imprimible {
  imprimir(): string;
}

class UsuarioBase {
  constructor(protected id: string) {}
}

class Admin extends UsuarioBase implements Imprimible {
  imprimir(): string {
    return `Admin ${this.id}`;
  }
}

Idea clave para no confundirte

Si solo necesitas compartir datos/funciones entre 2-3 clases cercanas, herencia simple está bien. Si la jerarquía crece, suele ser más mantenible componer objetos en lugar de encadenar muchas clases hijas.

Sobrescritura de métodos y super

Igual que en demo.-js/16-ES6-herencia.js: puedes redefinir un método y reutilizar parte del comportamiento base.

class Person {
  greet(): void {
    console.log("Hola");
  }
}

class Admin extends Person {
  override greet(): void {
    super.greet();
    console.log("Soy admin");
  }
}

Composición (preferida en muchos casos)

El proyecto también ilustra composición en la factura (Invoice contiene Company y Product).

class Product {
  constructor(public name: string, public price: number) {}
}
class Invoice {
  constructor(public product: Product, public amount: number) {}
}

4Objetos, funciones constructoras y prototipos

Las clases de ES6 son sintaxis más cómoda sobre prototipos. Cuando un método no está en el objeto, JavaScript lo busca en su prototipo.

  • Los datos propios viven en cada instancia.
  • Los métodos comunes viven en prototype y se comparten.
function Persona(this: any, nombre: string) {
  this.nombre = nombre;
}
(Persona as any).prototype.saludar = function () {
  return `Hola ${this.nombre}`;
};

Traducción mental: "cada persona guarda su nombre, pero todas comparten la misma función saludar".

Sombra de propiedades (shadowing)

En demo.-js/10-proto.js se ve: si escribes una propiedad con el mismo nombre, creas una local en la instancia y "tapas" la heredada.

const base = { teeth: 32 };
const user = Object.create(base);

user.teeth; // 32 (heredada)
user.teeth = 30;
user.teeth; // 30 (propia)
Object.hasOwn(user, "teeth"); // true

Cadena de prototipos en herencia

Cuando haces class Hija extends Base, JS conecta automáticamente los prototipos (similar a Object.setPrototypeOf(Hija.prototype, Base.prototype) del ejemplo con funciones constructoras).

5Accessors: getters y setters

Los accessors permiten exponer una propiedad como si fuera simple, pero con lógica interna de lectura/escritura.

  • get: controla cómo se lee un valor.
  • set: valida o transforma antes de guardar.
  • Son útiles para proteger invariantes (por ejemplo, "el saldo nunca puede ser negativo").
class Cuenta {
  private _saldo = 0;

  get saldo(): number {
    return this._saldo;
  }

  set saldo(valor: number) {
    if (valor < 0) throw new Error("Saldo invalido");
    this._saldo = valor;
  }
}

Uso mental rápido

Desde fuera usas cuenta.saldo como propiedad normal. Por dentro, la clase decide si acepta o no el nuevo valor.

Estilo recomendado: en validaciones con error (throw) usa if por claridad; reserva el ternario para asignaciones simples.
