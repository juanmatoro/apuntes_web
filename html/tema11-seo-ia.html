11.1¿Qué es GEO (Generative Engine Optimization)?

GEO es el proceso de optimizar contenido para que sea visible y citado por buscadores basados en inteligencia artificial: ChatGPT Search, Perplexity, Google AI Overviews, Microsoft Copilot y otros motores generativos.

SEO tradicional vs GEO

Aspecto SEO Tradicional GEO / LLM SEO
Objetivo Posicionarse en la SERP Ser citado en respuestas de IA
Cómo funciona Indexación + ranking por algoritmo RAG: recuperación + generación
Señal principal Backlinks y palabras clave Claridad, estructura y autoridad
Formato preferido Keywords en títulos y texto Respuestas directas, tablas, listas
Frescura Importa, pero tolera contenido antiguo Señal de alto peso: los LLMs prefieren contenido reciente

Dato clave: Solo el 20% de las citas en AI Overviews de Google coincide con el resultado #1 tradicional. Posicionarse en Google no garantiza ser citado por la IA.

11.2¿Cómo leen tu HTML los buscadores de IA?

Los sistemas de búsqueda basados en IA usan RAG (Retrieval-Augmented Generation): primero recuperan fragmentos de páginas web relevantes y luego los usan como contexto para generar una respuesta.

El proceso paso a paso

  1. Crawling: El bot de IA (OAI-SearchBot, Bingbot, Googlebot) solicita el HTML de tu página
  2. Parsing: Elimina boilerplate (menús, footers, ads) y extrae el contenido principal
  3. Tokenización: Divide el texto en tokens y genera embeddings vectoriales
  4. Indexación vectorial: Almacena los fragmentos en una base de datos vectorial
  5. Recuperación: Cuando un usuario pregunta, busca los fragmentos más relevantes
  6. Generación: El LLM sintetiza la respuesta usando esos fragmentos como fuente

Implicación para tu HTML

Los modelos tienen una ventana de contexto limitada (budget de tokens). Si tu HTML está lleno de markup redundante, menús repetitivos o JavaScript pesado, el contenido importante puede quedar fuera del presupuesto de tokens y nunca ser procesado.

11.3HTML semántico como interfaz para la IA

Para un crawler de IA, el HTML semántico funciona como una API bien tipada: cada etiqueta le dice al modelo exactamente qué tipo de contenido está leyendo. Sin semántica, el modelo trata tu contenido como datos malformados y lo descarta.

Mal: «div soup»

<div class="wrapper">
  <div class="top-bar">...</div>
  <div class="content">
    <div class="title">Cómo usar HTML</div>
    <div class="text">HTML es...</div>
  </div>
</div>

La IA no puede distinguir qué es título, qué es contenido o qué es navegación.

Bien: HTML semántico

<header><nav>...</nav></header>
<main>
  <article>
    <h1>Cómo usar HTML</h1>
    <p>HTML es...</p>
  </article>
</main>
<footer>...</footer>

La IA sabe que <main> es el contenido principal y <h1> es el tema.

Jerarquía de encabezados para IA

Los LLMs usan la jerarquía de encabezados como mapa del contenido. Una buena práctica es convertir los encabezados en las mismas preguntas que la página responde:

<h1>Guía completa de Flexbox en CSS</h1>

<h2>¿Qué es Flexbox?</h2>
<p>Flexbox es un modelo de layout que...</p>
  <!-- Respuesta directa inmediatamente debajo -->

<h2>¿Cómo se usa justify-content?</h2>
<p>La propiedad justify-content distribuye...</p>

<h2>¿Cuál es la diferencia entre Flexbox y Grid?</h2>
<p>Flexbox es unidimensional mientras que Grid...</p>

11.4Estructurar contenido para la extracción por IA

Los motores generativos extraen fragmentos de tu página, no la página completa. Tu objetivo es que cada fragmento sea autocontenido y útil por sí mismo.

11.4.1 Patrón pregunta-respuesta

Coloca una respuesta directa y breve inmediatamente después del encabezado. Luego amplía con detalles.

<h2>¿Cuánto tarda en cargar una web rápida?</h2>
<p><strong>Una web rápida carga en menos de 2.5 segundos
(LCP).</strong> Este es el umbral recomendado por Google
para las Core Web Vitals...</p>

11.4.2 Tablas y listas para datos comparativos

Los LLMs procesan muy bien datos tabulados y listas. Son formatos ideales para comparaciones y pasos:

<!-- Tabla comparativa: fácil de extraer por IA -->
<table>
  <thead>
    <tr><th>Formato</th><th>Compresión</th><th>Soporte</th></tr>
  </thead>
  <tbody>
    <tr><td>WebP</td><td>Excelente</td><td>97%</td></tr>
    <tr><td>AVIF</td><td>Superior</td><td>92%</td></tr>
    <tr><td>JPEG</td><td>Buena</td><td>100%</td></tr>
  </tbody>
</table>

<!-- Lista de pasos: fácil de citar -->
<ol>
  <li>Instalar Node.js desde nodejs.org</li>
  <li>Ejecutar <code>npm init</code></li>
  <li>Instalar dependencias con <code>npm install</code></li>
</ol>

11.4.3 Datos con atribución clara

Los LLMs valoran estadísticas y datos con fuente explícita. Incluye la fuente junto al dato:

<!-- Bien: dato con fuente clara -->
<p>El tráfico referido por IA creció un 527% entre enero
y mayo de 2025
(<cite>Growth Memo, Kevin Indig</cite>).</p>

<!-- Mal: dato sin fuente -->
<p>El tráfico de IA ha crecido mucho.</p>

11.5Datos estructurados optimizados para IA

JSON-LD le da a los sistemas RAG un mapa semántico del contenido. Es especialmente útil para que la IA entienda relaciones entre entidades: autor, tema, organización, fechas.

11.5.1 Conectar entidades con @id

Crea un grafo de conocimiento interno conectando tus schemas con referencias @id:

<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@graph": [
    {
      "@type": "Organization",
      "@id": "https://ejemplo.com/#org",
      "name": "Mi Empresa",
      "url": "https://ejemplo.com"
    },
    {
      "@type": "Person",
      "@id": "https://ejemplo.com/#autor",
      "name": "Ana García",
      "worksFor": { "@id": "https://ejemplo.com/#org" }
    },
    {
      "@type": "Article",
      "headline": "Guía de HTML para principiantes",
      "author": { "@id": "https://ejemplo.com/#autor" },
      "publisher": { "@id": "https://ejemplo.com/#org" },
      "datePublished": "2025-10-15",
      "dateModified": "2026-01-20"
    }
  ]
}
</script>

11.6Configuración técnica para crawlers de IA

11.6.1 robots.txt para bots de IA

Asegúrate de permitir el acceso a los crawlers de IA más importantes. Muchos sitios bloquean estos bots sin saberlo:

# robots.txt - Permitir bots de IA
User-agent: Googlebot
Allow: /

User-agent: Bingbot
Allow: /

User-agent: OAI-SearchBot      # ChatGPT Search
Allow: /

User-agent: ChatGPT-User       # ChatGPT live browsing
Allow: /

User-agent: PerplexityBot
Allow: /

User-agent: ClaudeBot           # Anthropic
Allow: /

11.6.2 El archivo llms.txt

Es un estándar emergente (propuesto, no oficial) que declara las políticas de acceso para IA. Se coloca en la raíz del dominio:

# /llms.txt - Ejemplo

# Nombre del sitio
> Mi Web de Apuntes

# Descripción
Apuntes del curso de elaboración de documentos web.
Cubre HTML, CSS y buenas prácticas.

# Contenido principal
- [Apuntes HTML](https://ejemplo.com/apuntes/html/)
- [Apuntes CSS](https://ejemplo.com/apuntes/css/)

# Sitemap
- [Sitemap XML](https://ejemplo.com/sitemap.xml)

Sobre llms.txt

Este estándar está en debate. Algunos expertos consideran que los bots de IA actualmente usan el HTML normal, no archivos especiales. Sin embargo, configurarlo no perjudica y puede ser útil a medida que el estándar madure.

11.6.3 Renderizado server-side

Los crawlers de IA no siempre ejecutan JavaScript. Si tu contenido depende de JS para renderizarse, puede que nunca sea indexado:

  • Prioriza HTML estático o SSR (Server-Side Rendering)
  • Usa <noscript> como fallback para contenido crítico
  • Mantiene el tiempo de respuesta del servidor bajo 200ms
  • Evita SPA (Single Page Apps) sin SSR para contenido público

11.7Optimizar el presupuesto de tokens

Los modelos de IA tienen una ventana de contexto limitada. Cada token de HTML innecesario es un token que no se usa para tu contenido real.

Sí hacer

  • HTML limpio y semántico
  • Contenido principal al inicio del body
  • Respuestas directas bajo los encabezados
  • CSS y JS en archivos externos
  • Contenido conciso y bien estructurado

No hacer

  • CSS inline masivo o bloques <style> largos
  • JavaScript inline extenso
  • Menús de navegación repetidos/largos
  • Divs anidados innecesariamente
  • Texto oculto o contenido duplicado

11.8E-E-A-T: Experiencia, Expertise, Autoridad y Confianza

Los LLMs priorizan contenido que demuestra expertise real. E-E-A-T se puede reforzar desde el HTML:

  • Experiencia: incluir ejemplos prácticos reales, casos de uso propios
  • Expertise: datos del autor con Schema Person, enlaces a perfil profesional
  • Autoridad: citas con fuentes, datos estructurados Organization
  • Confianza: HTTPS, política de privacidad, contenido actualizado con dateModified
<!-- Autor visible en la página -->
<address class="author">
  Escrito por <a rel="author" href="/about/ana">Ana García</a>,
  desarrolladora web con 10 años de experiencia.
  <time datetime="2026-01-15">Actualizado: 15 enero 2026</time>
</address>

<!-- Schema.org para el autor -->
<script type="application/ld+json">
{
  "@type": "Person",
  "name": "Ana García",
  "jobTitle": "Desarrolladora Web Senior",
  "sameAs": [
    "https://linkedin.com/in/ana-garcia",
    "https://github.com/anagarcia"
  ]
}
</script>

11.9Checklist GEO para tu HTML

Estructura y contenido

  • □ HTML semántico (header, main, article, section, footer)
  • □ Encabezados como preguntas que la página responde
  • □ Respuesta directa bajo cada <h2>
  • □ Tablas y listas para datos comparativos
  • □ Estadísticas con fuente explícita (<cite>)
  • □ Contenido actualizado con frecuencia
  • □ Lenguaje claro y directo (evitar metáforas)

Técnico

  • □ robots.txt permite bots de IA
  • □ JSON-LD con @graph conectando entidades
  • □ Datos del autor visibles + Schema Person
  • □ dateModified en el Schema
  • □ HTML renderizado server-side
  • □ Tiempo de respuesta < 200ms
  • □ XML sitemap actualizado y segmentado
