9.1¿Qué es la accesibilidad web?

La accesibilidad web (a11y) garantiza que todas las personas, incluidas aquellas con discapacidades visuales, auditivas, motoras o cognitivas, puedan percibir, entender, navegar e interactuar con la web.

Principios POUR (WCAG 2.1)

  • Perceptible: la información debe ser presentable de formas que el usuario pueda percibir
  • Operable: la interfaz debe poder manejarse con distintos dispositivos (teclado, ratón, voz...)
  • Comprensible: el contenido y la navegación deben ser fáciles de entender
  • Robusto: el contenido debe ser interpretable por una amplia variedad de tecnologías, incluyendo lectores de pantalla

Niveles de conformidad WCAG

Nivel Descripción Ejemplo
A Mínimo, requisitos básicos Texto alternativo en imágenes
AA Recomendado, estándar legal en la UE Contraste de color mínimo 4.5:1
AAA Óptimo, el más exigente Contraste 7:1, lenguaje de signos

9.2HTML semántico como base de la accesibilidad

Usar las etiquetas HTML correctas es la forma más efectiva de mejorar la accesibilidad. Los lectores de pantalla y tecnologías de asistencia dependen de la semántica del HTML para interpretar el contenido.

Mal: HTML no semántico

<div class="header">
  <div class="nav">
    <span onclick="ir()">Inicio</span>
  </div>
</div>
<div class="main">
  <div class="titulo">Artículo</div>
  <div class="texto">Contenido...</div>
</div>

Bien: HTML semántico

<header>
  <nav>
    <a href="/">Inicio</a>
  </nav>
</header>
<main>
  <article>
    <h1>Artículo</h1>
    <p>Contenido...</p>
  </article>
</main>

Regla de oro

Si existe un elemento HTML nativo que describe lo que quieres hacer, úsalo. No reinventes con <div> y <span> lo que ya existe como <button>, <nav> o <a>.

9.3Atributos ARIA (Accessible Rich Internet Applications)

ARIA proporciona atributos adicionales para mejorar la accesibilidad cuando el HTML semántico no es suficiente. Se divide en tres categorías:

9.3.1 Roles ARIA

Definen qué tipo de elemento es para las tecnologías de asistencia.

Rol Propósito Equivalente HTML nativo
role="banner" Cabecera principal del sitio <header>
role="navigation" Bloque de navegación <nav>
role="main" Contenido principal <main>
role="alert" Mensaje urgente para el usuario No tiene equivalente directo
role="button" Elemento que actúa como botón <button>

9.3.2 Propiedades ARIA

Dan información adicional sobre un elemento. No cambian con la interacción del usuario.

  • aria-label: etiqueta invisible para lectores de pantalla
  • aria-labelledby: referencia al id de otro elemento que actúa como etiqueta
  • aria-describedby: referencia a un elemento que da una descripción más detallada
  • aria-required="true": indica que un campo es obligatorio
  • aria-placeholder: texto indicativo cuando no hay placeholder nativo
<!-- aria-label: botón sin texto visible -->
<button aria-label="Cerrar menú">
  <img src="icono-x.svg" alt="">
</button>

<!-- aria-labelledby: reutilizar un título existente -->
<h2 id="seccion-contacto">Contacto</h2>
<form aria-labelledby="seccion-contacto">
  ...
</form>

<!-- aria-describedby: mensaje de ayuda -->
<label for="pass">Contraseña</label>
<input type="password" id="pass" aria-describedby="pass-help">
<small id="pass-help">Mínimo 8 caracteres</small>

9.3.3 Estados ARIA

Reflejan condiciones dinámicas que cambian con la interacción del usuario.

  • aria-expanded="true|false": indica si un elemento desplegable está abierto
  • aria-hidden="true": oculta un elemento de los lectores de pantalla
  • aria-disabled="true": indica que un elemento está deshabilitado
  • aria-selected="true": indica la opción seleccionada actualmente
  • aria-live="polite|assertive": anuncia cambios dinámicos en el contenido
<!-- Menú desplegable accesible -->
<button aria-expanded="false" aria-controls="menu-principal">
  Menú ▼
</button>
<ul id="menu-principal" aria-hidden="true">
  <li><a href="/">Inicio</a></li>
  <li><a href="/about">Acerca de</a></li>
</ul>

<!-- Zona de notificaciones en vivo -->
<div aria-live="polite" aria-atomic="true">
  <!-- Contenido actualizado por JS será anunciado -->
</div>

9.4Imágenes accesibles

El atributo alt es fundamental pero su contenido debe adaptarse al contexto:

Tipo de imagen Atributo alt Ejemplo
Informativa Describir el contenido alt="Gráfico de ventas Q3 2024: +15%"
Decorativa Vacío (alt="") alt="" (separadores, fondos)
Funcional (dentro de enlace) Describir la acción alt="Ir a la página de inicio"
Compleja (gráficos, diagramas) Resumen breve + descripción larga Usar <figure> + <figcaption> 
<!-- Imagen compleja con figure -->
<figure>
  <img src="organigrama.png"
       alt="Organigrama de la empresa con 4 departamentos">
  <figcaption>
    La empresa se organiza en: Dirección, Desarrollo,
    Marketing y Soporte. Cada departamento reporta
    directamente a Dirección.
  </figcaption>
</figure>

<!-- Imagen decorativa: alt vacío -->
<img src="separador.svg" alt="" role="presentation">

9.5Formularios accesibles

Los formularios son una de las áreas más críticas para la accesibilidad. Cada campo debe estar correctamente etiquetado y los errores deben comunicarse de forma clara.

Reglas clave

  • Siempre usar <label for="id">
  • Agrupar campos relacionados con <fieldset> y <legend>
  • Indicar campos obligatorios con texto, no solo color
  • Mensajes de error vinculados con aria-describedby
  • Usar autocomplete en campos comunes
  • Orden lógico de tabulación (no manipular tabindex innecesariamente)

Ejemplo interactivo

Datos personales



Nunca compartiremos tu email 

<form>
  <fieldset>
    <legend>Datos personales</legend>

    <label for="nombre">Nombre <span aria-hidden="true">*</span></label>
    <input type="text" id="nombre" required
           autocomplete="given-name"
           aria-required="true">

    <label for="email">Email <span aria-hidden="true">*</span></label>
    <input type="email" id="email" required
           autocomplete="email"
           aria-required="true"
           aria-describedby="email-error">
    <span id="email-error" role="alert">
      <!-- Se rellena con JS si hay error -->
    </span>
  </fieldset>
</form>

9.6Navegación por teclado

Todo elemento interactivo debe ser alcanzable y operable únicamente con el teclado.

Tecla Acción esperada
TabMover al siguiente elemento interactivo
Shift+TabMover al anterior
EnterActivar enlace o botón
EspacioActivar botón, marcar checkbox
EscCerrar modal/menú
FlechasNavegar dentro de listas, tabs, radios
 
<!-- Enlace "Saltar al contenido" para usuarios de teclado -->
<body>
  <a href="#contenido-principal" class="skip-link">
    Saltar al contenido principal
  </a>
  <header>...navegación larga...</header>
  <main id="contenido-principal" tabindex="-1">
    ...contenido...
  </main>
</body>

<!-- CSS para el skip-link -->
<style>
.skip-link {
  position: absolute;
  top: -40px;
  left: 0;
}
.skip-link:focus {
  top: 0; /* Solo visible al recibir foco */
}
</style>

9.7Herramientas de validación

Extensiones de navegador

  • axe DevTools: auditoría automática de accesibilidad
  • WAVE: informe visual de errores
  • Lighthouse: integrado en Chrome DevTools
  • HeadingsMap: visualiza la jerarquía de encabezados

Tests manuales imprescindibles

  • Navegar toda la página solo con Tab
  • Verificar que el foco es visible en todo momento
  • Probar con un lector de pantalla (NVDA, VoiceOver)
  • Hacer zoom al 200% y verificar que nada se rompe
  • Desactivar CSS y comprobar que el contenido tiene sentido

Checklist rápido de accesibilidad

  • □ Todas las imágenes tienen alt apropiado
  • □ Todos los inputs tienen <label>
  • □ Contraste de color ≥ 4.5:1 (texto normal) / 3:1 (texto grande)
  • □ Se puede navegar completamente con teclado
  • □ La jerarquía de encabezados es lógica (h1 → h2 → h3)
  • □ El atributo lang está definido en <html>
  • □ Los enlaces tienen texto descriptivo (no «clic aquí»)
  • □ Los vídeos tienen subtítulos
