1Client-Side Rendering (CSR)

Vamos a construir una pequeña app de tareas (Todo) formada completamente por componentes JS. Para ello usamos CSR: el servidor entrega un HTML vacío y JavaScript construye y actualiza toda la interfaz en el navegador.

SSR — Server-Side Rendering

  • El servidor genera el HTML completo.
  • El navegador recibe HTML ya renderizado.
  • Cada navegación recarga la página.
  • Menor control dinámico desde el cliente.

CSR — Client-Side Rendering

  • El servidor entrega un HTML mínimo (shell).
  • JS construye la UI en el navegador.
  • Las actualizaciones no recargan la página.
  • Permite componentes reutilizables y estado.

Flujo CSR

1. Navegador pide la página → 2. Servidor devuelve HTML vacío + <script> → 3. JS monta los componentes en los placeholders → 4. Eventos actualizan el estado → re-render

El HTML shell de nuestra app

Todo lo que el servidor entrega son unos placeholders vacíos y el <script>. Cada placeholder tiene un atributo data-component que JS usará para localizar el hueco y sustituirlo por el componente real.

<!-- index.html: solo el esqueleto, sin contenido visible -->
<body>
  <div data-component="header"></div>   <!-- JS colocará el Header aquí -->
  <div data-component="form"></div>     <!-- JS colocará el Form aquí -->
  <div id="list-root"></div>             <!-- contenedor estable para la lista -->
  <script src="app.js"></script>        <!-- JS construye todo lo anterior -->
</body>
Clave: en CSR la fuente de verdad es el estado en memoria (un objeto JS), no el HTML. Cuando el estado cambia, JS regenera el HTML del componente afectado y lo vuelve a pintar.

2Modelo de componente — Header

El primer componente que montamos es el Header. Es el más sencillo: se pinta una sola vez y nunca cambia. El patrón es siempre el mismo: crear el nodo, estilarlo, inyectarle HTML con datos y sustituir el placeholder.

Placeholder en index.html

<!-- hueco vacío; JS lo sustituye al cargar -->
<div data-component="header"></div>

El atributo data-component actúa como identificador. JS lo localiza con querySelector.

Componente en app.js

// 1. DEFINICIÓN: construye el nodo, no lo inserta
function Header(app) {

  // createElement crea el elemento en memoria (no en la página)
  const el = document.createElement('header');

  // className asigna las clases CSS al nodo
  el.className = 'bg-blue-700 text-white p-4 text-xl';

  // innerHTML rellena el interior con datos recibidos como parámetro
  el.innerHTML = `<h1>${app.titulo}</h1>`;

  return el;  // devuelve el nodo listo (aún fuera del DOM)
}

// 2. MONTAJE: localiza el placeholder y lo sustituye por el nodo
const ph = document.querySelector('[data-component="header"]');
ph.replaceWith(Header({ titulo: 'Mis Tareas' }));
Principio clave: el componente no sabe dónde va a vivir. Solo construye el nodo y lo devuelve. Es el código de montaje (fuera de la función) quien decide dónde insertarlo con replaceWith.

3Render y composición — TodoList

La lista de tareas cambia con cada acción del usuario, por lo que necesita re-renderizarse. Usamos dos componentes anidados: TodoItem genera el HTML de una tarea; TodoList los compone todos y los inyecta en su contenedor.

A diferencia del Header (que usa createElement + replaceWith), aquí los componentes devuelven strings HTML que se asignan vía innerHTML al contenedor estable #list-root, lo que permite re-pintar la lista entera de forma sencilla.

// TodoItem: recibe un objeto todo y devuelve su HTML como string.
// Se compone dentro de TodoList, no necesita createElement.
function TodoItem(todo) {
  // data-id almacena el id en el DOM para poder borrarlo después
  return `<li class="flex justify-between p-2 border-b" data-id="${todo.id}">
    <span>${todo.texto}</span>
    <button data-id="${todo.id}">Eliminar</button>
  </li>`;
}

// TodoList: compone varios TodoItem y devuelve HTML string.
// Cada llamada genera el HTML completo y actualizado de la lista.
function TodoList(todos) {
  if (todos.length === 0) return '<p class="text-gray-400">Sin tareas.</p>';
  return `<ul>${todos.map(TodoItem).join('')}</ul>`;
}

// renderList: inyecta la lista en su contenedor estable (#list-root).
// Se llama cada vez que el estado cambia para actualizar la vista.
function renderList(todos) {
  document.getElementById('list-root').innerHTML = TodoList(todos);
}
Composición: TodoList no sabe cómo se pinta cada tarea; delega ese detalle en TodoItem. Cada componente tiene una única responsabilidad.

4Estado y actualización

El estado es el único objeto que contiene los datos de la app. setState aplica cualquier cambio sobre él y acto seguido llama a renderList y bindEvents para que la vista refleje el nuevo estado.

// Estado central: fuente de verdad de la app
const state = { todos: [] };

// setState: mezcla los cambios en el estado y actualiza la vista.
// Nunca se modifica state directamente desde fuera de esta función.
function setState(cambios) {
  Object.assign(state, cambios);  // aplica los cambios: state.todos = [...]
  renderList(state.todos);        // pinta la lista con los nuevos datos
  bindEvents();                   // re-enlaza eventos al HTML recién generado
}

// Arranque: montamos los componentes estáticos y pintamos la lista inicial
document.querySelector('[data-component="header"]')
  .replaceWith(Header({ titulo: 'Mis Tareas' }));

document.querySelector('[data-component="form"]')
  .replaceWith(Form());   // Form es otro componente (input + botón Añadir)

renderList(state.todos);  // lista vacía al inicio
bindEvents();             // prepara los listeners desde el primer render
Regla de oro: el DOM es una consecuencia del estado, nunca al revés. Nunca modificar el DOM directamente; siempre pasar por setState para mantener estado y vista sincronizados.

5Flujo CRUD básico

Cada operación modifica el estado (nunca el DOM directamente) y setState se encarga del re-render. bindEvents debe llamarse tras cada render porque el HTML nuevo no conserva los listeners del anterior.

Operaciones sobre el estado

// CREATE: añade una tarea nueva al array
function addTodo(texto) {
  const todo = { id: crypto.randomUUID(), texto };
  setState({ todos: [...state.todos, todo] });
}

// READ: ya implementado en renderList + TodoList

// DELETE: filtra el array eliminando el id
function deleteTodo(id) {
  setState({ todos: state.todos.filter(t => t.id !== id) });
}

// UPDATE: sustituye el item modificado (mismo patrón)
function updateTodo(id, texto) {
  setState({
    todos: state.todos.map(t => t.id === id ? { ...t, texto } : t)
  });
}

Enlace de eventos

// bindEvents: asigna listeners al HTML recién generado.
// Se llama tras cada setState porque innerHTML destruye
// el HTML anterior junto con sus listeners.
function bindEvents() {

  // Botones "Eliminar" de cada tarea
  document.querySelectorAll('[data-id]').forEach(btn => {
    btn.addEventListener('click', () => deleteTodo(btn.dataset.id));
  });

  // Botón "Añadir" del formulario
  document.querySelector('#btn-add')
    ?.addEventListener('click', () => {
      const input = document.querySelector('#input-todo');
      if (input.value.trim()) {
        addTodo(input.value.trim());
        input.value = '';  // limpiar el campo tras añadir
      }
    });
}
Por qué re-enlazar eventos: cada vez que se asigna innerHTML, los nodos anteriores desaparecen y con ellos sus listeners. bindEvents siempre trabaja sobre el HTML recién generado.

6Buenas prácticas

  • Funciones puras para componentes: mismos datos → mismo HTML. Sin efectos secundarios dentro del componente.
  • Un solo estado central: nunca modificar el DOM directamente; siempre pasar por setState.
  • Separar capas: estado / componentes (HTML) / montaje / eventos. Cada capa tiene una responsabilidad.
  • Re-renderizar solo lo que cambia: en nuestra app solo actualizamos #list-root, no toda la página.
  • Escalar a framework (React, Vue, Svelte...) cuando el número de componentes o la lógica de estado crezcan demasiado.
